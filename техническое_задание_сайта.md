# Техническое Задание на Разработку Сайта

## Общие положения

Данный документ описывает техническое задание на создание веб-сайта с использованием чистого HTML и CSS.

## 1. Структура Сайта

Сайт должен иметь следующую базовую структуру:

-   **Основное содержимое (Main Content Area):** Область для отображения содержимого выбранной страницы.
-   **Боковая панель (Sidebar):** Левая панель навигации.
-   **Футер (Footer):** Не указан в текущем ТЗ, возможно будет добавлен позже.

## 2. Боковая Панель (Sidebar)

### 2.1. Заголовок Sidebar

-   В верхней части боковой панели должен располагаться заголовок с текстом: **"Ломоносов"**.

### 2.2. Элементы Навигации (Кнопки/Ссылки)

Боковая панель должна содержать следующие интерактивные элементы (кнопки или ссылки) для навигации по страницам сайта:
-   **Лабораторные**
-   **Практические**
-   **Рефераты**
-   **Практические страницы**
-   **Сайты**

### 2.3. Внешний Вид и Поведение

-   Боковая панель должна быть изначально скрыта (например, показывая только часть для индикации) и полностью выдвигаться при наведении курсора мыши (hover).
-   Стилизация боковой панели и её элементов должна быть выполнена с использованием чистого CSS.
-   При нажатии на любой из навигационных элементов, соответствующая страница должна загружаться в основную область контента.

## 3. Технологии

-   **Фронтенд:** Чистый HTML5 и CSS3. Использование JavaScript на данном этапе не предполагается, если иное не будет указано в последующих итерациях ТЗ.

## 5. Бэкенд (Backend)

### 5.1. Технологии
-   **Среда выполнения:** Node.js.
-   **Каркас:** Express.js для обработки HTTP-запросов и маршрутизации.
-   **Формат данных:** JSON.

### 5.2. Структура Хранения Данных
Данные о работах и сайтах будут храниться в JSON-файлах в специальной директории (например, `/data/`).

-   `data/laboratory_works.json`
-   `data/practical_works.json`
-   `data/essays.json`
-   `data/sites.json`

### 5.3. Структура JSON-объектов

**Для `laboratory_works.json`, `practical_works.json`, и `essays.json`:**
Массив объектов, где каждый объект представляет одну работу.
```json
[
  {
    "id": 1,
    "name": "Лабораторная работа №1",
    "date": "15.01.2026",
    "file_link": "/uploads/lab_report_1.docx"
  }
]
```

**Для `sites.json`:**
Массив объектов, где каждый объект представляет один сайт.
```json
[
  {
    "id": 1,
    "name": "Портфолио",
    "site_link": "https://example.com",
    "figma_link": "https://figma.com/file/..."
  }
]
```

**Для `kursovie_works.json`:**
Массив объектов, где каждый объект представляет одну курсовую работу.
```json
[
  {
    "id": 1,
    "name": "Курсовая работа по предмету",
    "date": "15.01.2026",
    "doc_file_link": "/uploads/kursovie_doc_1.docx",
    "zip_file_link": "/uploads/kursovie_product_1.zip"
  }
]
```

### 5.4. API (Интерфейс Программирования Приложений)

Бэкенд должен предоставлять следующие эндпоинты:

-   **`POST /api/works/:type`**
    -   **Описание:** Добавление новой работы.
    -   `:type` может быть `laboratory`, `practical`, `essays`, `otrabotka`, `narabotka`, или `kursovie`.
    -   **Тело запроса:** JSON-объект с полями `{ "name", "file_link", "password" }`. Для `kursovie` типа, ожидаются поля `{ "name", "doc_file_link", "zip_file_link", "password" }`. Дата (`date`) генерируется сервером.
    -   **Действие:** Бэкенд должен добавить новый объект в соответствующий JSON-файл.
    -   **Ответ:** Статус успеха или ошибки.

-   **`GET /api/sites`**
    -   **Описание:** Получение списка всех сайтов.
    -   **Ответ:** Массив объектов из `sites.json`.

-   **`POST /api/sites`**
    -   **Описание:** Добавление нового сайта.
    -   **Тело запроса:** JSON-объект с полями `{ "name", "site_link", "figma_link", "password" }`.
    -   **Действие:** Бэкенд должен добавить новый объект в `sites.json`.
    -   **Ответ:** Статус успеха или ошибки.

-   **`DELETE /api/works/:type/:id`**
    -   **Описание:** Удаление работы по ID.
    -   `:type` может быть `laboratory`, `practical` или `essays`.
    -   `:id` - идентификатор удаляемой работы.
    -   **Тело запроса:** JSON-объект с полем `{ "password" }`.
    -   **Действие:** Бэкенд должен удалить объект с указанным `id` из соответствующего JSON-файла.
    -   **Ответ:** Статус успеха или ошибки.

-   **`DELETE /api/sites/:id`**
    -   **Описание:** Удаление сайта по ID.
    -   `:id` - идентификатор удаляемого сайта.
    -   **Тело запроса:** JSON-объект с полем `{ "password" }`.
    -   **Действие:** Бэкенд должен удалить объект с указанным `id` из `sites.json`.
    -   **Ответ:** Статус успеха или ошибки.

### 5.5. Клиент-сайд Логика (JavaScript)

-   При загрузке соответствующей страницы (Лабораторные, Практические, Рефераты, Сайты), JavaScript-код должен отправлять `GET` запрос к API для получения актуального списка карточек.
-   Полученные данные должны динамически отрисовываться на странице в виде HTML-карточек.
-   Каждая карточка должна содержать кнопку "Удалить".
-   При нажатии на кнопку "Добавить" или "Удалить" приложение должно запрашивать у пользователя пароль.
-   После отправки формы (для добавления) или при нажатии на кнопку удаления, JavaScript должен выполнить `POST` или `DELETE` запрос к соответствующему эндпоинту API, передавая в теле запроса введенные пользователем данные и пароль.
-   После успешного добавления или удаления, страница должна автоматически обновиться, чтобы отобразить актуальный список карточек.
-   Должна быть реализована возможность загрузки файлов (`.docx`, `.pdf`) на сервер, которые будут сохраняться в директорию (например, `/uploads/`), а ссылка на них будет записываться в JSON-файл.
